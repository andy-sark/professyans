# 05 · Roadmap: задачи для Cursor

Пред-разбитые атомарные задачи для следующих итераций. Каждая секция — это *серия* задач, которые можно последовательно скармливать Cursor'у. Формулировки уже близки к prompt-ready; перед отправкой дополни их файловыми reference-ами (`@filename`).

Используй шаблоны из `04_CURSOR_PLAYBOOK.md`.

## Итерация 4: Формула-5

F-5 (2005) — упрощённая версия F-7 (45 карт, 5 групп, без СЧЖ/К/ВК). См. спеку §6.

### Задача 4.1. Данные F-5

**Один промпт для Cursor:**

Создать `shared-data/formula5/cards.json` по образцу F-7 cards.json с 45 карточками и meta. Формулировки — из `professyans_spec_v2.md` §6.3. Ключевые отличия от F-7:

- Ц-2 — «Исследовать что-либо, делать открытия» (НЕ «Торговать»).
- О-3 — «Возможность выпивать и ругаться на работе» (НЕ «Минимальные ограничения»).
- Нет групп СЧЖ, К, ВК.
- Формула — 8 карточек (не 7), минимум одна из каждой из 5 групп.
- `meta.mainGroups` = все 5 групп (отличие от F-7, где только 6 из 8).

Acceptance: JSON валидный, 45 уникальных `code`, 9 карт в каждой группе.

### Задача 4.2. Core F-5

Создать `core/src/professyans_core/methods/formula5.py`. Структура — как `formula7.py`, но:

- `F5.formula_size = 8`.
- `F5.main_groups` = все 5 групп.
- Переиспользовать функции `validate_formula` и пр. в generic-виде: если в `formula7.py` логика завязана на `F7`-константы, извлечь generic-функции в `methods/common.py` и вызвать из обоих файлов.
- Добавить `shared-data/formula5/hints.json` со стартовыми ~10 сигнатурами F-5 (это отдельная задача с психологом; в MVP — 5-7 очевидных, помеченных как `"draft": true` в meta).

Acceptance: `cd core && pytest` проходит; тесты F-7 не сломались.

### Задача 4.3. Frontend F-5

Создать по аналогии с F-7:
- `frontend/src/data/formula5/cards.ts` — обёртка над JSON.
- `frontend/src/lib/f5/validation.ts`, `f5/hints.ts`.
- `frontend/src/screens/formula5/F5Intro.tsx`, `F5Ranking.tsx`, `F5FormulaBuild.tsx`, `F5MoleculeBuild.tsx`, `F5Results.tsx` — можно смотреть на F-7 как эталон, но не копировать слепо: у F-5 нет СЧЖ/ВК-шагов, молекула из 8 карт, нет сверки с life-style.
- Роуты в `App.tsx`.
- В `HomeScreen.tsx::METHODS` — поставить `available: true` для F-5.

Acceptance: `npm run typecheck` + `npm run build` проходят; ручной прогон F-5 до экрана результатов работает в dev.

### Задача 4.4. Backend F-5

В `backend/app/api/sessions.py::get_result` добавить обработку `method == "F5"` (вызвать `core.methods.formula5.derive_result`). Integration-тест в `backend/tests/test_api.py`.

Acceptance: `cd backend && pytest` проходит; curl-тест POST F-5-сессии + GET /result возвращает корректные hints.

## Итерация 5: КЧГ («Кто? Что? Где?»)

См. спеку §7. 84 карточки, 3 группы (К-27, Ч-27, Г-27), уникальная матрица связей: на каждой К-карточке указаны номера Ч и Г, в которых это качество проявляется.

### Задача 5.1. Данные КЧГ

Создать `shared-data/kchg/cards.json`. Структура meta — `method: "KCHG"`, `groups` = [К, Ч, Г]. Главная особенность:

- К-карточки (27 штук) имеют поля `linked_Ch: number[]` и `linked_G: number[]` — массивы номеров карточек Ч и Г.
- Ч- и Г-карточки не имеют этих полей.

Формулировки карточек — из спеки §7.2. Матрица связей — там же, в таблицах «К × Ч», «К × Г». Это ~500+ связей, нужна внимательная транскрипция. Не выдумывать, строго сверять.

Acceptance: 84 карточки, все К-карты имеют непустые `linked_Ch` и `linked_G`.

### Задача 5.2. Core КЧГ

Создать `core/src/professyans_core/methods/kchg.py`. Ключевые функции:

- `rank_qualities(selected_k: list[str]) -> list[QualityRank]` — из выбранных К ранжировать по количеству связей.
- `compute_match_matrix(selected_k, selected_ch, selected_g) -> MatchMatrix` — автоматическая сверка (см. спеку §7.3). Формат: процент совпадения К↔Ч и К↔Г.
- `derive_kchg_result(session) -> KCHGResult` — высокоуровневый.

Acceptance: тесты в `core/tests/test_kchg.py`, parity с планируемой frontend-логикой.

### Задача 5.3. Frontend КЧГ

Три этапа работы пользователя: К → Ч → Г. После каждого — ранжирование + выбор N карточек.

Ключевой компонент — экран результатов с *матрицей совпадения*. Должно быть визуально ясно:
- какие выбранные К поддерживаются выбранными Ч;
- какие выбранные К поддерживаются выбранными Г;
- процент совпадения (*с дисклеймером, что это «очень приблизительные» подсказки* — прямая цитата из спеки §7.1).

Acceptance: ручной прогон от экрана интро до матрицы работает.

## Итерация 6: Перекрёсток (опросник)

См. отдельную спеку `perekrestok_spec_v2.md`. Это *не* карточная методика, а опросник на 222 вопроса с подсчётом по сферам/средствам/образованию/самостоятельности и сведением через таблицы пересечений.

### Задача 6.1. Данные опросника

Создать `shared-data/perekrestok/`:

- `questions.json` — 222 вопроса в 4 подчастях (P1.1, P1.2, P2.1, P2.2). Каждый: `{id, part, text, isReverse, rowIndex, colIndex, note?}`.
- `spheres.json` — 20 сфер с нормами для юношей и девушек (из спеки §16).
- `means.json` — 5 групп средств + нормы.
- `education_levels.json` — 5 уровней + нормы.
- `autonomy_levels.json` — 7 уровней + нормы.
- `intersections.json` — таблица «предметы × средства труда», 20×5 = 100 ячеек, в каждой — список профессий.
- `education_autonomy.json` — таблица «образование × самостоятельность», 5×7 = 35 ячеек.

Это большая задача. Разбить на под-задачи по файлу. Источник — спека §15–16. ⚠️ PDF-источник содержит OCR-ошибки, нужна ручная сверка с бумажной книгой (ВАКО, 2005).

### Задача 6.2. Core Перекрёсток

`core/src/professyans_core/methods/perekrestok.py`:

- `score_part(answers) -> PartScore` для каждой из 4 подчастей.
- `compute_intersections(sphere_tops, means_tops) -> list[Profession]` — извлечение из таблицы пересечений.
- `compute_education_autonomy_match(...)`.
- `derive_perekrestok_result(session) -> PerekrestokResult`.

Особенности: нормы разные для гендеров (обязательный параметр session.gender); «обратные» вопросы (isReverse=true) инвертируются при подсчёте.

### Задача 6.3. Frontend Перекрёсток

Длинный опросник — особый UX:

- Ответ на шкале 0–5 (кнопки, не slider).
- Честный прогресс-бар (спека §3.2: «вопрос 47 из 120»).
- Автосохранение после каждого ответа.
- Предложение перерыва каждые 30 вопросов.
- Разделение на 4 сессии (по подчастям) опционально.
- Экран выбора гендера перед началом (обязательный, нельзя пропустить).

Экраны:
- `PIntro.tsx` — введение, выбор трека, выбор гендера.
- `PQuestionnaire.tsx` — цикл вопросов с прогрессом.
- `PPartSummary.tsx` — сводка после подчасти + провокация.
- `PResults.tsx` — финал с таблицей пересечений и профессиями.

## Итерация 7: Авторизация и режим консультанта

См. спеку `professyans_spec_v2.md` §15.

### Задача 7.1. Auth infrastructure

- Backend: JWT-based auth, bcrypt для паролей, эндпоинты /auth/register, /auth/login, /auth/me.
- Frontend: auth store в Zustand, ProtectedRoute, страница /login.
- Соблюдать опциональность: базовый сценарий работает без аккаунта.

### Задача 7.2. Role: consultant

Добавить роль «консультант» с возможностью:
- видеть список клиентов, их session_id и статус;
- открывать session клиента в read-only режиме (с трекингом процесса на полную);
- оставлять комментарии, которые клиент видит.

Особенности:
- Клиент должен явно открыть доступ консультанту через код (одноразовая ссылка или 6-значный код).
- Доступ отзывается в любой момент.
- Read-only — консультант не меняет данные, только читает и комментирует.

### Задача 7.3. Live-режим

WebSocket или Server-Sent Events для того, чтобы консультант видел активность клиента в реальном времени (в рамках активной session). MVP — polling раз в 5 секунд, вполне достаточно.

## Итерация 8: Desktop (dmg/exe)

### Задача 8.1. PyWebView oboлочка

Создать `desktop/` пакет:

- Python-main: поднимает встроенный uvicorn с FastAPI (те же `core` и `backend/app/`, но с SQLite вместо Postgres).
- PyWebView открывает окно с `frontend/dist/index.html`, проксируя API на `http://127.0.0.1:<port>`.
- Конфигурация: `DATABASE_URL=sqlite:///{user_data_dir}/professyans.db`.

### Задача 8.2. Сборка

PyInstaller-spec:
- macOS (dmg): `.app`-бандл с иконкой, подписью (если будет cert).
- Windows (exe): `.exe` + installer через NSIS или Inno Setup.

Shared-data и frontend/dist — включаются в бандл как asset-каталоги.

### Задача 8.3. Автообновление (опционально)

Проверка версии на публичном endpoint, опциональное скачивание apдейта. Для MVP — без авто-обновления, просто показать ссылку «новая версия доступна».

## Итерация 9: PDF-экспорт, i18n, a11y

### Задача 9.1. PDF-экспорт

Библиотека `pdfmake` на клиенте. Экспорт:
- Формулы (молекула).
- История процесса.
- Список напряжений.
- Открытые вопросы.

Все данные уже в сессии — нужно только форматирование. Спека §4.4: не встроенная реклама/брендинг, нейтральный layout.

### Задача 9.2. i18n

Язык-пакеты: `ru` (основной), `uk` (украинский), `en`.

Сложность: формулировки карточек — это *content*, а не UI-строки. Перевод карточек — отдельная задача с психологом. В MVP переводим только UI (меню, кнопки, дисклеймеры).

### Задача 9.3. A11y-аудит

- Контрастность AAA для body text.
- Клавиатурная навигация во всех флоу.
- ARIA-labels на всех интерактивных элементах.
- Скринридер-тест (NVDA/VoiceOver) сквозного F-7.
- Сайз-регулировка — кнопки «увеличить/уменьшить шрифт» в header.

## Общий процесс работы по задаче

Когда берёшь задачу из этого roadmap:

1. **Прочитай** соответствующий раздел спеки (через `06_SPEC_MAP.md`).
2. **Создай feature-branch**: `git checkout -b feat/f5-cards-data` (см. `07_GIT_WORKFLOW.md`).
3. **Напиши промпт** для Cursor по шаблону из `04_CURSOR_PLAYBOOK.md`.
4. **Проверь** результат: тесты, typecheck, ручной прогон.
5. **Обнови документацию** если добавилось новое (карточки → `02_DATA_CONTRACT.md`, конвенции → `03_CONVENTIONS.md`).
6. **Commit** по формату из `07_GIT_WORKFLOW.md`.
7. **Обнови CHANGELOG.md**.
8. **Merge** в main (или соответствующий вашему процессу branch).
