# 01 · Архитектура: рационал решений

Этот документ объясняет *почему* проект устроен именно так. Цель — предотвратить решения, которые технически работают, но ломают методологию.

---

## Почему монорепо

Четыре пакета (`frontend/`, `backend/`, `core/`, `shared-data/`) связаны контрактами типов и данных, которые должны эволюционировать синхронно. В раздельных репозиториях это быстро деградирует до «Фронт обновили, бэк забыли, теперь поля разные» — классический провал микросервисов на MVP-стадии.

Монорепо даёт:
- одну команду `docker compose up` для полного стека;
- атомарные коммиты «меняю JSON + TS-типы + Python-модели»;
- общий CI-пайплайн в будущем;
- `shared-data/` как артефакт, видимый всем пакетам.

Когда имеет смысл выделить отдельный репо: если появляется сторонняя организация (например, мобильное приложение под iOS), которой не нужен весь монорепо. Пока такого нет — держим вместе.

---

## Почему `core/` отделён от `backend/`

`core/` — это *pure-Python-библиотека* без IO и HTTP. Никаких FastAPI, SQLAlchemy, uvicorn. Только бизнес-логика методик + модели данных.

**Причина:** desktop-сборка (PyInstaller → dmg/exe, следующая итерация) будет включать то же ядро без FastAPI. Единственная разница — слой хранения (SQLite вместо Postgres) и UI-оболочка (PyWebView вместо браузера).

Если бы логика жила внутри `backend/`, пришлось бы либо тащить FastAPI в desktop-билд, либо дублировать код. Первое утяжеляет бинарь на 30+ МБ, второе ломает parity.

**Правило:** в `core/` не должно быть `import fastapi`, `import sqlalchemy`, `from app import ...`. Если добавляешь зависимость — проверь, что она не тянет за собой http/db-инфраструктуру.

---

## Почему JSON в `shared-data/` — canonical

Три силы тянут в противоположные стороны:

1. **Фронт** хочет typed imports и bundle-time данные.
2. **Бэк** хочет server-side доступ + cache-friendly endpoints.
3. **Психолог-редактор** хочет править формулировки без git-push и перекомпиляции.

Единственный формат, удовлетворяющий всем трём — JSON в отдельной папке:
- Фронт импортирует через Vite alias `@shared-data` (типы из TS-wrapper'ов в `frontend/src/data/formula7/*.ts`).
- Бэк читает через `professyans_core.paths.load_json()`.
- Редактор открывает в любом JSON-аware редакторе (VS Code, онлайн-CMS) и правит.

**Критично:** это означает, что JSON — *source of truth*, а TS и Python код вокруг него — derivative. Никогда не правь данные в TS или Python напрямую; только в JSON.

Для добавления нового поля схема такая: сначала JSON → затем TS-wrapper (обновить интерфейс) → затем Python-модель → затем тесты. См. `03_CONVENTIONS.md` §«Правило parity».

## Почему offline-first + opt-in sync

Методика активизирующая; пользователь может прерваться, вернуться через неделю, передумать и перезапустить. Онлайн-зависимость — это фрикция, которая:
- ломает UX когда интернет плохой;
- пугает пользователей с тревогой приватности (опросник — интимная штука);
- требует авторизации ради сохранения, что меняет психологический контракт методики.

Решение: **primary storage = localStorage + IndexedDB на устройстве**. Синхронизация с backend — *дополнительная галочка* на главной, которая выключена по умолчанию. Если backend недоступен, фронт продолжает работать как будто бэка нет.

Это уже реализовано в `frontend/src/lib/api.ts` — клиент никогда не бросает исключений; ошибки сети пишутся в console.warn и игнорируются.

**Антипаттерн:** делать авторизацию обязательной для старта сессии. Методика требует, чтобы порог входа был нулевым.

## Почему два трека: «активизирующий» и «закрытый»

Прямая цитата автора (спек §4.7): «автор методики — против упрощённого теста. Но мы понимаем: иногда нужно быстро. Выбор за тобой.»

Два трека — это *этический компромисс*:
- **Активизирующий** — полная методика с провокациями, трекингом процесса, разбором противоречий, пятой фазой с открытыми вопросами.
- **Закрытый** — быстрый проход до списка профессий, с явным дисклеймером «это упрощение, не полная картина».

Экран выбора трека *сам по себе* активизирующий элемент — он честно называет trade-off и делает пользователя соучастником решения.

**Никогда** не делать трек по умолчанию «активизирующий» автоматически — выбор должен быть явным.

---

## Почему пятая фаза критична

Из спеки §4.3: «программа, которая не выдержала тон пятой фазы, разрушает всю предыдущую работу».

Пятая фаза — это экран результатов. Её UX-требования жёсткие:
- молекула как *карта* (визуально), не как список;
- история процесса (колебания, возвраты) показывается как *зеркало*, не как оценка;
- напряжения подсвечиваются, но программа их *не исправляет*;
- вывод в виде *открытых вопросов*, не диагноза;
- формулировки типа «ваш тип личности», «рекомендуемые профессии:» — запрещены.

Реализация: `frontend/src/screens/formula7/F7Results.tsx`. Любые изменения этого экрана требуют перепроверки против спек §4.3–4.6 и §16.

## Почему parity TS ↔ Python

Две реализации одной логики — это плохо *вообще*, но у нас это осознанный выбор:

- **Фронт** считает всё локально, потому что offline-first.
- **Бэк** считает всё server-side, потому что desktop и будущий режим консультанта нуждаются в серверной агрегации.

Единственная альтернатива — сделать расчёты только на бэке и требовать онлайн. Это ломает offline-first. Другая альтернатива — WASM-модуль с Rust/Go. Избыточно для MVP.

**Тест на parity:** `backend/tests/test_api.py::test_get_result_computes_hints_and_conflicts` — PUT-ит сессию и проверяет, что server-side вернёт тот же hint (`h4` IT) с теми же `matchedKeys`.

Если расходится — это ошибка, не опциональная фича. См. `03_CONVENTIONS.md` §«Правило parity» для процесса.

## Почему Zustand, а не Redux

Стейт сессии — это один объект `Session`, который меняется действиями пользователя и персистится в localStorage. Redux с его store/reducers/actions/selectors — overkill. Zustand даёт тот же результат в 100 строк кода без boilerplate.

Когда имеет смысл переходить: если появится сложная междукомпонентная синхронизация, time-travel debugging в production, или несколько независимых глобальных состояний. Пока всё в одном `useSession` — Zustand идеален.

---

## Почему SQLAlchemy с JSON-blob, а не normalized tables

`SessionRecord` хранит основные поля (id, method, track, started_at, ...) как индексированные колонки, а весь остальной стейт — как JSON-блоб в колонке `state`.

**Плюсы:**
- Фронт сериализует `Session` в JSON и PUT-ит на сервер без трансформации.
- Бэк десериализует и возвращает — тоже без маппинга.
- Схема Session может эволюционировать без миграций.
- Postgres JSONB + GIN index даёт хороший поиск по вложенным полям, когда понадобится.

**Минусы:**
- Нельзя делать аналитику обычным SQL (но для активизирующей методики это и не планируется).
- Нет foreign keys на уровне ORM к карточкам (но это OK — карточки живут в JSON-данных, не в БД).

Когда имеет смысл нормализовать: если появится режим консультанта с агрегированной статистикой по клиентам, и нужны будут быстрые запросы «все клиенты, у которых Ц-6 в формуле». Тогда добавим отдельную таблицу `formula_cards` как материализованную view.

## Деплой-топология (план)

```
┌─────────────────┐      HTTPS      ┌──────────────────┐
│  Browser (SPA)  │ ──────────────► │ nginx (frontend) │
│  or PyWebView   │                  │  static assets    │
└─────────────────┘                  └──────────────────┘
        │                                     
        │ HTTPS (opt-in sync)                 
        ▼                                     
┌─────────────────┐                  ┌──────────────────┐
│  FastAPI        │ ◄──SQLAlchemy──► │  PostgreSQL 16   │
│  (uvicorn)      │                  │                  │
└─────────────────┘                  └──────────────────┘

Desktop variant (PyInstaller → dmg/exe):
┌───────────────────────────────────────┐
│ PyWebView + embedded FastAPI + SQLite │
│ (все процессы в одном бинаре)          │
└───────────────────────────────────────┘
```

В MVP — всё локально (`docker compose up`). Production — отдельный VPS с nginx как reverse proxy (TLS, rate-limiting), managed Postgres (DigitalOcean/Neon/Supabase).

## Что менять нельзя (красные линии)

Решения, которые *принципиальны* и не должны переобсуждаться без весомой причины:

1. **Методика не диагностическая.** Никаких личностных типов, никаких «ваших идеальных профессий», никаких процентов соответствия.
2. **Авторизация не обязательна** для базового использования.
3. **Провокации не гасить.** Формулировки автора могут казаться резкими — это часть методики, не баг.
4. **Правило «хотя бы одна из каждой группы»** в Формуле-7 — жёсткое, блокирующее переход. Это авторский инвариант.
5. **Нормы для юношей и девушек разные** в Перекрёстке — методологический выбор автора 2005 года, не гендерное предубеждение. Усреднять без явного запроса нельзя.
6. **Трекинг процесса** не опциональная фича, а ядро методики. Удалять его в угоду производительности — нельзя.
