# 04 · Playbook промптов для Cursor

Этот документ — для будущего Claude, который пишет промпты для Cursor по этому проекту. Цель: максимизировать полезность Cursor'а и минимизировать количество итераций.

---

## Как Cursor работает в этом проекте

Cursor — это IDE с AI-агентом. Он автоматически видит:

1. **`AGENTS.md`** в корне (cross-tool стандарт под Linux Foundation, читают Cursor/Codex/Copilot/Windsurf/Amp/Devin/Jules).
2. **`.cursor/rules/*.mdc`** — project rules с четырьмя режимами активации:
   - **Always** (`alwaysApply: true`) — всегда в контексте.
   - **Auto Attached** (`globs: [...]`) — автоматически подгружается при правке файлов по glob-паттерну.
   - **Agent Requested** (только `description`) — агент сам решает, нужно ли, по описанию.
   - **Manual** — только при `@rule-name` упоминании.
3. **Открытые файлы** в редакторе.
4. **Файлы, переданные через `@`-reference** в промпте.

Cursor НЕ видит:
- Файлы из `.cursorignore` (секреты, локальные БД) — вообще никогда.
- Файлы из `.cursorindexingignore` (lockfiles, кеши, dist/) — только если явно не открыты.
- Git-историю, issues, внешние ссылки.
- Что-либо вне checkout репозитория. Спеки `professyans_spec_v2.md` и `perekrestok_spec_v2.md` лежат в корне репо — доступ через `@professyans_spec_v2.md`.

Вывод: **промпт должен включать всё необходимое по `@`-reference**. Не рассчитывай, что Cursor «помнит» предыдущий разговор или «догадается» о конвенциях.

### Что где лежит (карта project rules)

| Файл | Режим | Когда применяется |
|---|---|---|
| `.cursor/rules/00-global.mdc` | always | **всегда** — красные линии, стек, parity, required reading |
| `.cursor/rules/10-architecture.mdc` | agent-requested | когда агент решит, что обсуждается архитектура |
| `.cursor/rules/20-data-contract.mdc` | auto-attached | при правке `shared-data/**`, `**/data/**`, `methods/**` |
| `.cursor/rules/30-parity.mdc` | auto-attached | при правке `frontend/src/lib/f7/**`, `core/src/professyans_core/methods/**` |
| `AGENTS.md` | always (по стандарту) | **всегда** — проектный README для агентов |

Новое правило добавляется как `.cursor/rules/<prefix>-<name>.mdc` — префикс по возрастанию (40-typescript.mdc, 50-python.mdc), чтобы было понятно читать.

## Анатомия хорошего промпта

Структура, которая работает стабильно:

```
### Цель
<Одно предложение: что должно появиться/измениться>

### Контекст
<2-4 предложения: ЗАЧЕМ это нужно, связь с методикой>

### Обязательное к чтению (reference)
- @docs/02_DATA_CONTRACT.md
- @frontend/src/screens/formula7/F7Ranking.tsx
- @shared-data/formula7/cards.json (первые 100 строк достаточно)

### Задача
1. <конкретный шаг>
2. <конкретный шаг>
3. <конкретный шаг>

### Ограничения
- <что нельзя делать>
- <какие правила соблюдать>

### Acceptance
- [ ] <проверяемый критерий>
- [ ] <проверяемый критерий>

### Если застрянешь

Каждый раздел короткий: одно-два предложения, максимум 5-7 пунктов. Слишком длинные промпты Cursor читает невнимательно.

## Шаблоны по типу задачи

### Шаблон 1: Итерация 5 — графическая молекула

```
### Цель
Подготовить переход с кнопочной молекулы на drag-and-drop полотно с эмерджентными ядрами для F-7 и F-5.

### Контекст
Текущий MoleculeBuild (кнопки 1/2/3) не передаёт методический смысл «карты, а не списка».
Нужно интерактивное размещение карточек и явные ядра, чтобы трекинг колебаний отражал реальный процесс.

### Обязательное к чтению
- @docs/05_ROADMAP_TASKS.md (итерация 5)
- @docs/06_SPEC_MAP.md (§4.3 и §10.4)
- @frontend/src/screens/formula7/F7MoleculeBuild.tsx
- @frontend/src/screens/formula5/F5MoleculeBuild.tsx
- @frontend/src/lib/tracker.ts

### Задача
1. Подготовить инфраструктуру dnd-kit (`@dnd-kit/core`, `@dnd-kit/utilities`) без изменения бизнес-логики results.
2. Ввести модель относительных координат карточек x/y ∈ [0..1] и стабильные id ядер.
3. Добавить базовые record-события для перемещений/создания/удаления ядра.
4. Обеспечить валидацию перехода к Results: минимум 2 ядра, минимум 1 карточка в каждом, максимум 5 ядер.
5. Сохранить parity TS↔Python на уровне структуры сессии.

### Ограничения
- НЕ ломать текущий кнопочный сценарий до полного завершения новой реализации.
- НЕ использовать абсолютные пиксели в персисте позиций (только относительные координаты).
- НЕ добавлять миграцию legacy-сессий: в PET-проекте допустим wipe IndexedDB.
- НЕ менять методические тексты в shared-data.

### Acceptance
- [ ] Карточку можно перетаскивать на мобильном и desktop.
- [ ] Позиции восстанавливаются после перезагрузки.
- [ ] Валидация ядер блокирует переход к Results при некорректной конфигурации.
- [ ] cd frontend && npm run typecheck и npm test проходят.

### Если застрянешь
- Если сомневаешься в методической интерпретации — сверяйся с `professyans_spec_v2.md` §4.3 и §10.4.
- Если UX деградировал на touch — сначала упрости interaction-модель, затем расширяй.
```

### Шаблон 2: Поправить UI-компонент

На экране F7Ranking сделать так, чтобы при прокрутке к последней карточке группы

### Acceptance
- [ ] Vite dev-сервер собирается без ошибок.
- [ ] npm run typecheck проходит.
- [ ] Визуально: при скролле на группе «СЧЖ» футер появляется после 3-й карточки.

## Антипаттерны промптов (чего НЕ делать)

### ❌ Слишком широкая задача

> «Доработай приложение, чтобы оно лучше соответствовало спеке»

Cursor переделает что попало. Всегда — конкретный файл, конкретный эффект.

### ❌ Без контекста «зачем»

> «Добавь в `HomeScreen.tsx` кнопку, выключающую трекинг процесса»

Cursor может сделать, но это *запрещено* (см. красные линии `01_ARCHITECTURE.md`). Без указания «зачем» он не проверит правомерность.

**Исправленный промпт:**
> «Контекст: пользователь в спеке §17.4 имеет право отказаться от показа инсайтов процесса (но не от самой записи — запись обязательна). Цель: добавить в HomeScreen toggle "показывать историю процесса на экране результатов", значение хранится в localStorage, читается F7Results.»

### ❌ Без acceptance-критериев

> «Обнови документацию по шаблону»

Cursor не знает, что такое «по шаблону» конкретно. Acceptance должен быть проверяемым: «все 120 тестов проходят», «vite build возвращает 0», «визуально работает в Chrome».

### ❌ Смешивать задачи

> «Добавь эндпоинт, поправь компонент, обнови тесты, и заодно отрефактори router»

Cursor часть сделает, часть забудет, часть сломает. Один промпт — одна атомарная задача. Рефакторинг — отдельно.

### ❌ Игнорирование конвенций

> «Напиши компонент кнопки с материальным дизайном»

У нас бумажно-травяной дизайн-язык, не Material. Всегда ссылайся на `03_CONVENTIONS.md` и существующие компоненты как эталон.

## Чеклист перед отправкой промпта

- [ ] Указана ли ОДНА атомарная цель?
- [ ] Есть ли ссылки через `@filename` на все нужные файлы?
- [ ] Есть ли связь с методологией (зачем это вообще)?
- [ ] Указаны ли запреты (чего не делать)?
- [ ] Есть ли проверяемые acceptance-критерии?
- [ ] Есть ли fallback «если застрянешь»?

Если хотя бы три галочки — промпт готов. Иначе — дорабатывай.

---

## Как интерпретировать ответ Cursor'a

После того как Cursor сделал изменения, всегда проверяй:

1. **Соответствие acceptance.** Галочки из промпта — выполнены?
2. **Не сломал ли существующее.** Запусти все тесты (`cd core && pytest`, `cd backend && pytest`, `cd frontend && npm test`, `cd frontend && npm run typecheck`).
3. **Не добавил ли антипаттерны.** Перечитай `01_ARCHITECTURE.md` §«Что менять нельзя».
4. **Не выдумал ли формулировки.** Если Cursor сочинил тексты карточек/провокаций — откатить. Формулировки только из спеки или в формате «расширить с психологом».

## Сценарии для новых чатов с Claude

Когда в новом чате с Claude ты говоришь «напиши промпт для Cursor», приложи этот стек:

1. Краткое описание задачи (3 строки).
2. Ссылка на `docs/00_INDEX.md` — чтобы Claude сориентировался.
3. Конкретный раздел `05_ROADMAP_TASKS.md`, если задача оттуда.
4. Актуальное состояние репозитория: `git log --oneline -20`, `git status`.

Claude увидит:
- `00_INDEX` → поймёт, где что;
- `04_CURSOR_PLAYBOOK` (этот файл) → применит шаблон;
- `05_ROADMAP_TASKS` → получит детализированную задачу.

И напишет Cursor-ready промпт за один заход, без лишних уточнений.
